# Трекер задач «Внешний мозг» — концепт и ТЗ для v1

## 1. Зачем это существует

Основная проблема пользователя — не нехватка времени, а когнитивная перегрузка от удержания планов в голове. Трекер должен работать как внешний мозг: задача выгружается из головы за секунды, а система сама решает, когда её показать. Инструмент проектируется под принцип «ленивой системы» — он не должен требовать ежедневных решений, дисциплины ведения или ручной пересборки при смене графика.

Ключевая особенность жизненного контекста: 5-дневная рабочая неделя с ротацией офис/удалёнка, которая меняется примерно раз в месяц (сейчас: пн–ср офис, чт–пт удалёнка). Вечера понедельника и вторника заняты клиентами частной практики. Зал — вечера удалённых дней (цель 3–4 раза в неделю: 3 силовые + 1 кардио). Дорога в офисные дни (~2 часа в день) — время для чтения по психоанализу. Поэтому вся логика планирования привязывается к **типам дней**, а не к дням недели: при смене ротации меняется только маппинг «дата → тип дня», и ничего больше.

## 2. Приоритеты v1 (в порядке важности)

1. **Автосборка плана «Сегодня»** — главная фича. Утром пользователь открывает приложение и видит готовый короткий план без единого решения с его стороны.
2. **Мгновенный ввод (inbox)** — задача попадает в систему за ≤5 секунд, одним текстовым полем.
3. **Дедлайны и напоминания** — жёсткие даты (запись к врачу, платежи по долгам) не теряются.
4. **Обзор недели по типам дней** — вторичное представление.

Интеграция с Google Calendar входит в v1.

## 3. Архитектура

Строим **API-first**, потому что в v2 планируются агенты и автоматизация: интерфейс — лишь один из клиентов API. Рекомендуемый стек (Claude Code может предложить эквивалентную замену, но принципы обязательны):

- **Бэкенд:** Node.js + Fastify (или Express), REST API с JSON. Вся бизнес-логика (включая сборку «Сегодня») живёт на сервере, не во фронтенде.
- **База:** SQLite (файл на диске) через better-sqlite3 или Drizzle ORM. Никакого localStorage для данных — только для UI-настроек.
- **Фронтенд:** React + Vite, одна SPA. Мобильная вёрстка обязательна (ввод задач чаще всего с телефона).
- **Google Calendar:** OAuth2 (Google Identity), токены хранятся на сервере. Библиотека googleapis.
- **Запуск:** локально одной командой (`npm run dev`), позже можно вынести на VPS/Raspberry Pi. Docker-файл — приятный бонус, не требование.

Задел под v2: все операции доступны через API с понятными эндпоинтами; в будущем поверх этого же API поднимется MCP-сервер для агентов. В v1 достаточно чистого REST и простой авторизации (единственный пользователь, токен в .env).

## 4. Модель данных

**Task**
- `id`, `title` (строка, единственное обязательное поле)
- `status`: `inbox` | `active` | `done` | `someday`
- `domain` (опционально): `pa_practice` | `fitness` | `piano` | `finance` | `health` | `life`
- `day_context` (опционально): `office` | `remote` | `weekend` | `any` — в какой тип дня задачу уместно делать
- `deadline` (опционально, дата) и `hard_deadline` (bool) — жёсткий дедлайн попадает в календарь
- `effort`: `quick` (<15 мин) | `normal` | `deep` (требует фокуса) — по умолчанию `normal`
- `week_flag` (bool) — «надо на этой неделе». Это ЕДИНСТВЕННЫЙ приоритет в системе. Никаких шкал 1–10.
- `created_at`, `completed_at`, `scheduled_date` (если задача явно назначена на дату)

**DayType** — конфигурируемый шаблон типа дня:
- `id`, `name` (`office`, `remote`, `weekend` — плюс возможность добавить свои)
- `slots`: описание доступных окон в JSON, например для office: `[{"name": "дорога", "capacity": "reading"}, {"name": "вечер", "capacity": "short", "busy_weekdays": [1, 2]}]` (пн/вт вечер занят клиентами)
- `daily_capacity`: сколько задач максимум собирать в «Сегодня» для этого типа дня (по умолчанию: office = 3, remote = 4, weekend = 5)

**Rotation** — маппинг «дата → тип дня»:
- Простая таблица правил вида «с 2026-07-01: пн,вт,ср = office; чт,пт = remote; сб,вс = weekend»
- При смене ротации добавляется новая запись с датой начала действия. Старые планы не пересобираются задним числом.

**Settings** — прочее: параметры GCal-синка, тихие часы для напоминаний и т.п.

## 5. Ядро: алгоритм автосборки «Сегодня»

Запускается при первом открытии приложения за день (и по запросу «пересобрать»). Шаги:

1. Определить тип сегодняшнего дня по Rotation.
2. Подтянуть события из Google Calendar на сегодня → вычислить реально свободные окна (клиенты, встречи, личные события уменьшают ёмкость дня).
3. Собрать кандидатов в порядке убывания важности: (а) задачи с дедлайном сегодня/просроченные, (б) задачи с `week_flag` и подходящим `day_context`, (в) задачи с `scheduled_date` = сегодня, (г) добивка из inbox — только `quick`-задачи, максимум одна-две.
4. Ограничить список `daily_capacity` этого типа дня. Лучше 3 выполненные задачи, чем 8 висящих.
5. Показать план + отдельным блоком события календаря. Пользователь может убрать/добавить задачу вручную — это исключение, а не норма.

Невыполненные задачи вечером НЕ «проваливаются» автоматически на завтра с чувством вины — они возвращаются в пул кандидатов и участвуют в сборке следующего подходящего дня. Никаких красных просроченных списков, кроме жёстких дедлайнов.

## 6. Интеграция с Google Calendar (объём v1)

- **Чтение:** события на сегодня и ближайшие 7 дней — для расчёта свободной ёмкости и отображения в «Сегодня»/«Неделя».
- **Запись:** только задачи с `hard_deadline` создаются событием (или событием на весь день) в отдельном календаре «Tracker». При изменении/выполнении задачи событие обновляется/удаляется.
- Двусторонний полный синк НЕ делаем — это болото. Календарь остаётся источником правды о встречах, трекер — о задачах.

## 7. Экраны

1. **Сегодня** (главный, открывается по умолчанию): дата и тип дня, план из 3–5 задач с чекбоксами, блок событий календаря, поле быстрого ввода внизу, кнопка «пересобрать».
2. **Inbox:** список неразобранных задач; свайп/кнопки для быстрой разметки (домен, контекст дня, week_flag) — разбор одной задачи ≤3 секунды.
3. **Неделя:** 7 колонок/строк с типом дня, дедлайнами и назначенными задачами; здесь же видно, сколько тренировок уже запланировано/сделано на неделе.
4. **Бэклог:** всё со статусом `someday` + фильтры по доменам. Открывается редко, и это нормально.
5. **Настройки:** типы дней, ротация, подключение Google, ёмкость дней.

Поле быстрого ввода доступно с любого экрана. Поддержать минимальный синтаксис в строке ввода: `!` = week_flag, `@дата` = дедлайн (например, «записаться к дантисту @15.07 !»). Парсер простой, без NLP — это кандидат на агентскую фичу в v2.

## 8. Напоминания (v1 — минимально)

Браузерные уведомления (Notification API) при открытом приложении: утреннее «план собран» и напоминание за день до жёсткого дедлайна. Push через сервис-воркеры и Telegram-бот — в v2. Жёсткие дедлайны и так дублируются в Google Calendar, у которого свои надёжные нотификации — в v1 этого достаточно.

## 9. Явный анти-скоуп v1 (не делать)

Подзадачи и проекты; теги свободной формы; оценки времени в минутах; статистика и графики продуктивности; полный двусторонний синк с календарём; мультипользовательность; drag-and-drop планирование по часам. Каждая из этих вещей увеличивает трение ввода или стоимость поддержки — против философии системы.

## 10. Дорожка v2 (для контекста, не реализовывать)

- MCP-сервер поверх REST API → Claude как агент: голосовой/чатовый ввод задач, еженедельный обзор с предложениями, авторазбор inbox.
- Telegram-бот для ввода и утреннего плана.
- Связка с хабит-трекером (piano daily, чтение ПА, зал) — общая база или API-мост.
- Умный парсинг естественного языка при вводе.

## 11. Порядок реализации для Claude Code

1. Скелет: сервер + SQLite + миграции, модели Task/DayType/Rotation, сид с текущей ротацией (пн–ср office, чт–пт remote).
2. REST API: CRUD задач, эндпоинт `GET /api/today` с алгоритмом сборки (сначала без календаря).
3. Фронтенд: экран «Сегодня» + быстрый ввод + Inbox.
4. Google OAuth + чтение событий → учёт ёмкости в сборке.
5. Запись hard-deadline задач в календарь.
6. Экраны «Неделя», «Бэклог», «Настройки», синтаксис быстрого ввода, уведомления.

Каждый шаг — рабочее состояние приложения. После шага 3 системой уже можно пользоваться.
